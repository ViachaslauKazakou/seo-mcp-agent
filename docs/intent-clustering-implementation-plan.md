# План реализации: сбор семантики по интентам через кластеризацию

## Цель
Построить в проекте двухэтапный пайплайн:
1. Sentence-BERT для быстрого семантического группирования и первичной интент-оценки.
2. Cross-Encoder для точной переоценки спорных случаев и финального интента.

## Где встраивать в текущий проект
- API: `src/seo_agent/api/routers.py`
- HF-инструменты: `src/seo_agent/tools/hf/`
- Модели БД: `src/db/models.py`
- Миграции: `src/db/migrations/versions/`
- Тесты: `tests/`

## Этап 1. Проектирование интентов и метрик
1. Зафиксировать taxonomy интентов:
   - informational
   - commercial
   - transactional
   - navigational
2. Для каждого интента описать краткие критерии и примеры.
3. Определить confidence-метрики:
   - на уровне ключа
   - на уровне кластера
4. Зафиксировать пороги ambiguous-кейсов (когда нужно подключать Cross-Encoder).

Результат: единые правила разметки и принятия решения по интенту.

## Этап 2. Схема данных и миграции
1. Расширить таблицу ключей полями:
   - intent_sbert
   - intent_cross
   - intent_final
   - intent_confidence
   - re_rank_score
2. Расширить таблицу кластеров полями:
   - primary_intent
   - secondary_intent
   - intent_distribution
   - intent_confidence
   - review_required
3. Добавить Alembic migration.

Результат: БД хранит все промежуточные и финальные решения по интентам.

## Этап 3. Stage-1: Sentence-BERT
Роль Stage-1:
- построение эмбеддингов ключей
- первичная кластеризация
- baseline определение интента

Шаги:
1. Подготовить набор интент-прототипов (phrases/templates) для каждого интента.
2. Считать косинусную близость ключа к интент-прототипам.
3. Назначить первичный интент и confidence.
4. Если margin между top1 и top2 ниже порога, пометить ключ как ambiguous.

Рекомендованные модели:
- `paraphrase-multilingual-mpnet-base-v2`
- `intfloat/multilingual-e5-base`

Результат: быстрый baseline по интентам на всем массиве ключей.

## Этап 4. Stage-2: Cross-Encoder
Роль Stage-2:
- точная переоценка только сложных случаев
- rerank/verification для конфликтных решений

Когда запускать:
1. Только для ambiguous ключей из Stage-1.
2. Для top-N ключей каждого кластера (контроль качества темы).
3. Для конфликтных случаев в кластерах с низкой связностью.

Тип скоринга:
- pair (keyword, intent prompt) -> relevance score

Результат: более точный интент в спорных кейсах при контролируемой стоимости инференса.

## Этап 5. Merge-логика итогового интента
Правила принятия решения:
1. Если Stage-1 confidence высокий и margin высокий -> берем SBERT.
2. Если ambiguous -> используем Cross-Encoder.
3. При конфликте:
   - если score Cross-Encoder выше порога -> intent_final = cross
   - иначе оставляем intent_final = sbert и ставим review_required

На уровне кластера:
1. primary_intent = weighted vote по ключам.
2. secondary_intent = второй интент, если его доля выше порога.
3. intent_confidence = агрегат confidence ключей.

Результат: прозрачная и воспроизводимая логика final intent.

## Этап 6. API и UI интеграция
API:
1. Добавить endpoint intent enrichment по website_id/run_id.
2. Вернуть в API поля intent_final, confidence, model_source.
3. В endpoint кластеров добавить primary_intent/secondary_intent.

UI:
1. На странице кластеров показывать интент-бейджи и confidence.
2. В модальном окне кластера показывать интент каждого ключа и источник решения.
3. Добавить фильтр по интентам.

Результат: интенты становятся видимыми и управляемыми в интерфейсе.

## Этап 7. Тестирование
Unit-тесты:
1. Stage-1 intent scoring.
2. Stage-2 reranking.
3. Merge-логика final intent.

Integration-тесты:
1. Импорт ключей -> кластеризация -> intent enrichment -> чтение через API.
2. Проверка стабильности результата на фиксированном наборе данных.

Датасет для валидации:
- 100-300 вручную размеченных ключей в вашей нише.

Результат: контролируемое качество и защита от регрессий.

## Этап 8. Производительность и эксплуатация
1. Кэшировать эмбеддинги по hash ключа.
2. Батчить инференс SBERT и Cross-Encoder.
3. Запускать Cross-Encoder асинхронно (job queue), не в критическом API пути.
4. Добавить fallback в SBERT-only режиме при ограничениях ресурсов.

Мониторинг:
- p95 latency
- доля ambiguous кейсов
- доля конфликтов между Stage-1 и Stage-2
- доля ручной проверки

Результат: стабильная работа в проде и предсказуемая стоимость вычислений.

## Роли моделей в системе
Sentence-BERT:
- быстрый и масштабируемый baseline
- кластеризация, поиск похожести, первичный интент

Cross-Encoder:
- более медленный, но точный в сложных случаях
- финальная валидация и коррекция пограничных решений
